## Decide first (design-level, worth their own session)

- [ ] **review-gate commit detection is at the wrong layer** - `.claude/hooks/review-gate.ps1:56`. The regex `^\s*git(\.exe)?\s.*\bcommit\b` fails in both directions (both live-reproduced):
  - Over-match: `git log --grep=commit`, `git config commit.gpgsign false`, `git rev-parse HEAD^{commit}` are classified as commits; with a dirty tree or stale stamp these read-only commands are denied with a misleading "re-run the security review" message.
  - Under-match: `GIT_COMMITTER_DATE=... git commit` bypasses BOTH layers (doesn't start with `git ` for settings.json's `if: Bash(git *)` filter, doesn't match the `^\s*git` anchor). Undocumented gap, worse than the two in CLAUDE.md: the per-part regex catches neither, and "keep git commit standalone" doesn't cover it.
  - Cheap fix: anchor on the subcommand token (`^\s*git(\.exe)?\s+commit\b` + allowance for global options) and allow optional `\w+=\S*\s+` env prefixes; add the env-prefix form to CLAUDE.md's known-gaps list.
  - Deeper fix (verified workable): a real git pre-commit hook in a tracked dir + one-time `git config core.hooksPath`, enforcing only when `CLAUDECODE=1` (confirmed set in Claude's Bash env) so the user's own terminal stays ungated. Fires on every commit regardless of spelling; also sees the true index at commit time, closing the documented `cherry-pick -n` state-snapshot gap. Residual: `--no-verify` (deliberate spelling, outside the forgetting-not-malice threat model). The PreToolUse hook could then be deleted; review-gate.ps1 shrinks to the hash compare.

- [ ] **config loading: dot-source .ps1 vs Import-PowerShellDataFile .psd1** - `sso-autofill.ps1:380`. Config is loaded by executing `config.local.ps1` then probing `Test-Path "variable:$name"` per required name. Every value in config.example.ps1 is a literal, so a `.psd1` + `Import-PowerShellDataFile` (available in PS 5.1) covers it: `$cfg.ContainsKey` validation, values namespaced in one object, and a data file cannot assign script variables or execute code - structurally preventing the `$DryRun`-clobber class (see mechanical item 1) inside a secret-handling script. Cost: template rename, `$OpVault` -> `$cfg.OpVault` rewrite, README/DESIGN/CLAUDE.md updates, migrate the existing config.local.ps1. The ACE argument is weak under this threat model (config.local.ps1 sits next to the script anyway); the param-clobber prevention is the real benefit. Decide: migrate, or keep dot-sourcing and just fix the doc row.

## Mechanical batch (one tidy session)

Top findings, most severe first:

- [ ] **DESIGN.md documents `DryRun` as a config value; a config `$DryRun` silently overrides the `-DryRun` switch** - `docs/DESIGN.md:177`. CONFIRMED by stub test: config.local.ps1 is dot-sourced into script scope AFTER param binding (`sso-autofill.ps1:380`), so a user who copies the table row and runs `.\sso-autofill.ps1 -DryRun` gets a LIVE run (real secrets typed) while believing R2 masking applies. The "run start (DryRun=True)" log line at :375 is written before the dot-source, so the log misleads too. The shipped template never defines `$DryRun` (config.example.ps1:10 states the correct rule; DESIGN.md:329 likewise). Fix: delete/correct the DryRun row at DESIGN.md:177; optionally snapshot/re-assert `$DryRun` after the dot-source in the script.
- [ ] **`op` stderr aborts the run under PS 5.1 EAP=Stop** - `sso-autofill.ps1:289` and `:302`. CONFIRMED with repro: with `$ErrorActionPreference='Stop'`, `& $OpPath ... 2>$null` promotes ANY stderr line to a terminating NativeCommandError even on exit 0 (e.g. an op CLI update notice after the Hello unlock kills the login run). The curated `$LASTEXITCODE` branches at :291/:303-304 are unreachable; real op failures surface as op's raw first stderr line in the popup instead of the remediation hint. `$PSNativeCommandUseErrorActionPreference` at :40 is a no-op ordinary variable on 5.1. Fails safe (abort precedes typing). Fix: capture stderr without triggering promotion (e.g. run via `cmd /c ... 2>nul`, or temporarily relax EAP around the two calls, or redirect to a file) and keep the exit-code branches authoritative.
- [ ] **verify hook reports SelfTest failures as exit 1, not 2 - feedback never reaches Claude** - `.claude/hooks/verify-ps1.ps1:28`. CONFIRMED with repro: a failing `-SelfTest` is an uncaught `throw` (the SelfTest block at sso-autofill.ps1:368-372 precedes the `try` at :374) -> stderr from the child; the hook's `$output = & powershell.exe ... 2>&1` under its own EAP=Stop (:7) throws NativeCommandError before the `$LASTEXITCODE` check (:30), and the hook dies with exit 1. PostToolUse only feeds exit-2 output back, so an edit that parses but breaks behavior lands silently - the exact case the hook exists for. Parse-error path (exit 2 at :23) unaffected. Fix: wrap the child invocation in try/catch (or relax EAP around it) and map any failure to exit 2 with the captured output.
- [ ] **Multiple-match abort reports a count but no window titles** - `sso-autofill.ps1:210-213`. The zero-match path lists `$seenTitles`; the multi-match throw carries nothing, so README's tuning loop (:137-139 "check sso-autofill.log ... adjust WindowTitleMatch") has no data. Multi-match is the known live case (2026-07-02: bare "Cisco Secure Client" matched csc_ui + the login window). Titles are already in `$hits` and already treated as log-safe one branch above; append them to the message.
- [ ] **`WindowTitleMatch` wildcards are interpreted, contradicting the documented substring contract** - `sso-autofill.ps1:200`. `$w.Title -notlike "*$TitleMatch*"` interpolates the config value unescaped; a value like `Cisco Client [Login]` never matches its own literal title (`[Login]` = character class) -> baffling "No matching window found" that lists the window in its own seen-titles. Latent (default has no metachars) and fails safe. Fix: `$w.Title.IndexOf($TitleMatch, [StringComparison]::OrdinalIgnoreCase) -lt 0` or `-notmatch [regex]::Escape($TitleMatch)`; config.example.ps1:43 documents "substring matched (case-insensitive)".
- [ ] **verify hook comment overstates coverage; config-contract drift passes green** - `.claude/hooks/verify-ps1.ps1:27` (and the repo CLAUDE.md "catches cross-file breakage" claim). `-SelfTest` returns before any config file is read (sso-autofill.ps1:368-372), so renaming/adding a name in the `$required` list (:382) without updating config.example.ps1 (or vice versa) verifies clean and dies at the next copy-from-template live run. Fix: scope the wording to "parse + worker unit suite", or add a SelfTest case that dot-sources config.example.ps1 in a child scope and asserts every `$required` name is defined (makes the contract machine-checked).
- [ ] **Stale "one Windows Hello prompt" claims** - `README.md:119` (dry-run pass criterion), `sso-autofill.ps1:28` (.NOTES), `docs/DESIGN.md:184/190`. The implementation makes two `op` calls and per CLAUDE.md every call re-prompts Hello (brief cache only), so two prompts are the normal outcome; DESIGN.md's findings/open items already acknowledge this. A user applying the README checklist fails a working setup. Fix: "one or two Hello prompts, depending on 1Password's auto-lock window" at all three sites.

## Reviewed and rejected (do not "fix")

- 15s self-closing popup in Show-FatalError: deliberate, documented at the site (:350-353), in DESIGN.md (:332-335), and compensated by the log; the old blocking MessageBox was the bug (2026-07-01 invisible hang holding secrets).
- `Write-Log -ErrorAction SilentlyContinue` (:50): correct - a throwing logger would mask the original error or abort mid-typing (lockout risk) over a diagnostic failure; no visible fallback exists from a hidden process.
- `.exe` normalization in the process matcher (:191): earns its keep - config.example.ps1 ships `.exe` forms while `Get-Process.ProcessName` never has them; removing it breaks the default config. The second self-test case is its regression test, not a duplicate.
- In-process `-SelfTest` in verify-ps1.ps1 (skip the double powershell spawn): the naive swap inverts the contract - `return` leaves `$LASTEXITCODE` null (clean edits report failure) and a SelfTest `throw` kills the hook with exit 1 (real failures stop being fed back). Savings ~0.2s; keep the spawn.
- Old-template config.local.ps1 hard-fails on missing `OpPath` (:382): working as designed - fail-visible with an actionable message; a silent `'op'` default would reintroduce the stale-PATH hotkey failure OpPath exists to surface.
