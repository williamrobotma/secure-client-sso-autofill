## Task B - Windows Hello latency (biggest felt win, zero code)

The 10-29s op+Hello phase dwarfs everything code can touch. Per 1Password's app-integration security model (verified against developer.1password.com 2026-07-09), a fresh process (every hotkey launch) MUST re-authorize - the app being unlocked does not waive the prompt - so the prompt itself stays; the target is its 20s+ startup latency, which is environmental (typical is 3-5s). All steps below are USER-run: each is either an op call (= a Hello prompt) or a 1Password app setting.

1. Split cold-start from prompt latency - in ONE PowerShell window run twice:
   `Measure-Command { & $OpPath vault list | Out-Null }` (use the full OpPath from config.local.ps1).
   -> verify: 1st call = Hello + cold start; 2nd call reuses the process-session (no prompt) = pure op/app overhead. 2nd ~1-2s means the prompt display is the problem; 2nd also slow means op<->app IPC (see step 2).
2. Keep the 1Password desktop app resident (tray) - op has to wake it otherwise.
   -> verify: re-measure step 1 with the app confirmed running; compare.
3. 1Password Settings -> Security -> enable "Show Windows Hello prompt automatically" if off.
   -> verify: prompt appears within ~1-2s of the hotkey on the next run.
4. If still slow: check AV real-time scanning of op.exe (WinGet Packages path) - exclusion is the user's call, not required.

Exit: Hello-to-typing consistently under ~5s, or the residual is accepted and documented in DESIGN.md as the floor. Explicitly rejected: service-account tokens / cached sessions to skip Hello - security downgrade, see DESIGN.md non-goals.

## Task C - closed-loop screen waits (kill the timers; gated on probe)

Goal: replace the open-loop `Start-Sleep $DelayAfter*` between screens with "wait until the NEXT screen's field actually exists", via Windows UI Automation (UIA) reads of the Cisco WebView2. Faster (advance the moment a screen is ready) and safer (a slow page load can no longer put the password on the wrong screen - the misfire-lockout risk goes away structurally).

Gate - probe first (probe-uia.ps1, read-only; written 2026-07-09):

1. User runs `.\probe-uia.ps1` in their own terminal, starts a connect, logs in BY HAND; pastes probe-uia.log back.
   -> verify: the log shows per-screen state blocks. Decision matrix:
   - Edit fields visible with IsPassword flag on screen 2 -> BUILD (spec below).
   - Only titles/Documents change, no fields -> title/Document-change waits are a weaker fallback; decide then whether the marginal gain justifies it.
   - "(empty)" tree on every screen -> managed UIA can't see the WebView2; STOP - the tuned timers are already near-optimal, close this task and record the finding in DESIGN.md.

Build spec (only if the probe says BUILD):

- Mechanism: after typing a field + Enter, poll (~150ms) the validated target window's UIA tree until the expected next element appears - password screen = an Edit with IsPassword; OTP screen = a non-password Edit reachable after it (exact signatures come from the probe log, which also times the real screen loads).
- Where: the code to replace is the three `Start-Sleep -Milliseconds $DelayAfter*` calls in sso-autofill.ps1's entry point (each follows a `Send-KeysToTarget` call); the new wait helper belongs with the R1 window helpers (near `Assert-Foreground`). Reuse probe-uia.ps1's element-walking approach as the reference implementation for the UIA reads.
- `DelayAfter*` config names stay but become timeout CAPS (fail-visible abort on expiry, same as today's R1 aborts), not sleeps. No new config names; config gets simpler to reason about, not bigger.
- Typing stays SendKeys; R1-R4 unchanged. UIA is used to WAIT and additionally to gate: only type the password when the focused field in the R1-verified window is IsPassword - a strict strengthening of R1, never a replacement.
- Probe stays in the repo as the diagnostic for future SSO page changes.
- Estimated diff: ~100 lines in sso-autofill.ps1 (one wait helper + call-site changes); self-test gets a read-only UIA case only if it can run against an always-present window (else manual test note).
- Security review + stamp + commit per the gate, as always.

Non-goals for this whole pass (rejected with reasons, do not re-open):

- Bypassing/skipping the per-run Windows Hello prompt (process-scoped by 1Password's design; skipping it = storing a live credential).
- UIA SetValue injection instead of SendKeys (brittle against the M365 JS form; changes the typing trust model for little gain).
- New dependencies or a rewrite in another language.
